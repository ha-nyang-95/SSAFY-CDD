## 일반 UI/UX 원칙

1. **데이터 중심의 명료성 (Data-First Clarity)**
    - **원칙:** 모든 디자인 요소는 `데이터를 명확하고 정확하게 전달`하는 것을 최우선 목표로 삼아야 합니다. `불필요한 장식은 최소화`하고, 3D 시각화 보고서나 균열 분석 데이터가 가장 먼저, 가장 명확하게 보이도록 레이아웃을 구성해야 합니다. 이는 '정밀성'을 중시하는 타겟 사용자의 핵심 요구사항을 만족시킵니다.
2. **효율성 중심의 워크플로우 (Efficiency-Driven Workflow)**
    - **원칙:** 안전 관리자와 엔지니어의 `작업 효율을 극대화하도록 인터페이스를 설계`해야 합니다. 자주 사용하는 기능(예: 보고서 생성, 실시간 모니터링 전환)에 빠르게 접근할 수 있어야 하며, 작업 흐름이 논리적이고 예측 가능해야 합니다. `불필요한 클릭이나 단계를 줄여 사용자가 최소한의 노력으로 목표를 달성할 수 있도록 지원`합니다.
3. **점진적 정보 공개 (Progressive Disclosure)**
    - **원칙:** `한 화면에 너무 많은 정보를 한꺼번에 노출하여 사용자를 압도해서는 안 됩니다.` 처음에는 가장 중요한 정보와 요약(전체 구조물 상태, 심각한 균열 수 등)을 보여주고, 사용자가 특정 영역을 선택하면 더 상세한 데이터(균열의 깊이/너비, LiDAR 데이터 등)를 볼 수 있도록 단계적으로 정보를 제공해야 합니다. 이는 사용자의 인지 부하를 줄여 복잡한 데이터를 더 쉽게 분석하도록 돕습니다.
4. **일관성을 통한 신뢰 구축 (Confidence through Consistency)**
    - **원칙:** 애플리케이션 전체에서 아이콘, 버튼 스타일, 용어, 레이아웃 등 `모든 디자인 요소의 일관성을 유지`해야 합니다. 이러한 일관성은 사용자가 시스템의 작동 방식을 쉽게 학습하고 예측하게 만들어 '신뢰성'과 '안전성'에 대한 믿음을 줍니다. 중요한 결정을 내리는 전문가들에게 예측 불가능한 인터페이스는 곧 시스템에 대한 불신으로 이어질 수 있습니다.

## 프론트엔드 개발의 핵심 원칙

### **1. 폴더 및 파일 구조화 기법**

프로젝트의 규모가 커지고 복잡해질수록 잘 설계된 폴더 구조는 유지보수성과 확장성의 핵심이 됩니다. 대표적인 세 가지 구조화 방법론의 개념과 장단점을 설명하겠습니다.

---

### **컴포넌트 기반 구조 (Component-Based Structure)**

이 방식은 UI를 독립적이고 재사용 가능한 부품, 즉 컴포넌트로 분리하여 관리하는 데 중점을 둡니다. 가장 대표적인 예시가 **아토믹 디자인(Atomic Design)**입니다.

- **핵심 개념 및 계층 구조:**
    - **Atoms (원자):** 더 이상 분해할 수 없는 가장 작은 UI 단위입니다. (예: `Button`, `Input`, `Label`, `Icon`) 이들은 자체적으로는 의미가 크지 않지만, 애플리케이션의 기본 스타일과 빌딩 블록을 형성합니다.
    - **Molecules (분자):** 여러 개의 원자를 결합하여 특정 목적을 가진 UI 단위입니다. (예: `Label` + `Input` + `Button` = 검색 폼) 분자는 이제 구체적인 기능을 수행하기 시작합니다.
    - **Organisms (유기체):** 분자와 원자를 조합하여 만들어진 더 복잡하고 독립적인 UI 섹션입니다. (예: `Header`, `Footer`, `ProductCardList`) 유기체는 웹사이트의 명확한 한 영역을 담당합니다.
    - **Templates (템플릿):** 여러 유기체와 분자를 모아 페이지의 레이아웃 구조를 정의합니다. 실제 콘텐츠가 없는 와이어프레임과 같습니다. (예: 블로그 포스트 템플릿, 상품 상세 페이지 템플릿)
    - **Pages (페이지):** 템플릿에 실제 콘텐츠(텍스트, 이미지 등)를 채워 넣어 사용자에게 보여지는 최종 페이지입니다.
- **대규모 애플리케이션에서의 장단점:**
    - **장점:**
        - **높은 재사용성:** 가장 작은 단위부터 체계적으로 조합하므로 컴포넌트 재사용이 극대화됩니다.
        - **일관성 유지:** 디자인 시스템을 구축하기 용이하며, 전체 애플리케이션의 UI/UX 일관성을 보장합니다.
        - **체계적인 개발:** 신규 개발자도 구조를 쉽게 이해하고 어디에 어떤 컴포넌트를 만들어야 할지 명확히 알 수 있습니다.
        - **독립적인 테스트:** 각 컴포넌트를 독립적으로 개발하고 테스트할 수 있습니다.
    - **단점:**
        - **과도한 추상화:** 작은 프로젝트에서는 불필요하게 구조가 복잡해질 수 있습니다. (Over-engineering)
        - **컨텍스트 분리:** 특정 기능을 수정할 때 관련된 파일들이 `atoms`, `molecules`, `organisms` 등 여러 폴더에 흩어져 있어 찾기 번거로울 수 있습니다.
        - **초기 학습 곡선:** 팀 전체가 아토믹 디자인의 개념을 이해하고 따라야 하므로 초기 도입 비용이 발생합니다.

---

### **기능 기반 구조 (Feature-Based / Domain-Based Structure)**

파일을 기술적 역할(`components`, `hooks`, `apis`)이 아닌, 애플리케이션의 기능 또는 도메인을 중심으로 구성하는 방식입니다.

- **파일 구성 방식:**
프로젝트의 특정 기능과 관련된 모든 파일(UI 컴포넌트, 상태 관리 로직, API 호출 함수, 테스트 코드 등)을 하나의 폴더에 모읍니다.
    
    `src/
    ├── features/
    │   ├── authentication/
    │   │   ├── components/
    │   │   │   ├── LoginForm.js
    │   │   │   └── SignupForm.js
    │   │   ├── hooks/
    │   │   │   └── useAuth.js
    │   │   ├── api/
    │   │   │   └── authApi.js
    │   │   └── index.js
    │   ├── user-profile/
    │   │   ├── components/
    │   │   │   └── ProfileCard.js
    │   │   ├── api/
    │   │   │   └── userApi.js
    │   │   └── index.js
    └── ...`
    
- **컴포넌트 기반 구조보다 유리한 경우:**
    - **응집도 높은 개발:** 특정 기능을 개발하거나 수정할 때, 관련된 모든 코드가 한곳에 모여 있어 컨텍스트 전환 비용이 적습니다. 개발자는 해당 기능 폴더 내에서 대부분의 작업을 완료할 수 있습니다.
    - **코드 탐색 용이성:** "사용자 프로필 수정" 기능과 관련된 파일을 찾을 때 `src/features/user-profile` 폴더만 확인하면 되므로 매우 직관적입니다.
    - **독립적인 기능 관리:** 기능별로 폴더가 분리되어 있어 특정 기능을 제거하거나 다른 프로젝트로 이식하기가 상대적으로 용이합니다. 이는 마이크로 프론트엔드 아키텍처로 확장할 때 좋은 기반이 됩니다.

---

### **계층형 아키텍처 (Layered Architecture)**

대규모 엔터프라이즈급 애플리케이션에서 선호되는 구조로, 코드를 여러 개의 논리적 계층(Layer)으로 분리하고 계층 간의 명확한 의존성 규칙을 설정하는 방식입니다. **피처 슬라이스 디자인(Feature-Sliced Design)**이 대표적인 예입니다.

- **계층별 역할:**
    - `src/app`: 애플리케이션의 최상위 계층. 전역 설정, 라우터, 공통 Provider 등을 설정합니다.
    - `src/pages`: 특정 경로에 매핑되는 페이지 단위의 컴포넌트. 여러 `widgets`와 `features`를 조합하여 페이지를 구성합니다.
    - `src/widgets`: 여러 `features`나 `entities`를 조합한 독립적인 UI 블록. (예: `SideBar`, `NewsletterForm`)
    - `src/features`: 사용자 상호작용을 포함하는 기능 단위. (예: `AddToCartButton`, `UserLogin`) 비즈니스 로직을 담고 있습니다.
    - `src/entities`: 핵심 비즈니스 개체(Entity). (예: `User`, `Product`, `Order`) 자체 로직은 거의 없고 데이터 모델과 이를 표현하는 순수 UI 컴포넌트로 구성됩니다.
    - `src/shared`: 모든 계층에서 재사용 가능한 공통 코드. (예: UI Kit, API 클라이언트, 유틸리티 함수, 공통 타입 정의)
- **계층 간 의존성 규칙:**
이 아키텍처의 핵심은 **단방향 의존성**입니다. 상위 계층은 하위 계층에 의존할 수 있지만, 하위 계층은 절대 상위 계층을 참조할 수 없습니다.
    
    **`app` → `pages` → `widgets` → `features` → `entities` → `shared`**
    
    예를 들어, `features` 계층은 `entities`와 `shared`를 가져와 사용할 수 있지만, `widgets`나 `pages`를 알지 못해야 합니다. 이 규칙은 코드 간의 결합도를 낮추고 순환 참조를 방지하여 예측 가능하고 유지보수하기 쉬운 거대한 코드베이스를 만듭니다.
    

---

### **2. 웹 개발의 기본 원칙**

프로페셔널 프론트엔드 개발자는 단순히 UI를 만드는 것을 넘어, 성능, 접근성, 유지보수성 등 다양한 품질 기준을 만족시키는 웹 애플리케이션을 구축해야 합니다.

---

### **코어 웹 바이탈 (Core Web Vitals) & 성능**

사용자 경험의 품질을 측정하는 구글의 핵심 지표입니다.

- **핵심 지표 설명:**
    - **LCP (Largest Contentful Paint):** **로딩 성능.** 뷰포트 내에서 가장 큰 이미지 또는 텍스트 블록이 렌더링되는 시간을 측정합니다. 사용자에게 "페이지가 의미 있게 로드되었다"는 인식을 주는 시점입니다. (권장: 2.5초 이내)
    - **FID (First Input Delay):** **상호작용성.** 사용자가 처음으로 페이지와 상호작용(클릭, 탭 등)했을 때, 브라우저가 해당 이벤트에 응답하기까지 걸리는 시간을 측정합니다. 사용자가 "사이트가 즉각 반응한다"고 느끼게 하는 지표입니다. (권장: 100ms 이내)
    - **CLS (Cumulative Layout Shift):** **시각적 안정성.** 페이지 로딩 중 발생하는 예기치 않은 레이아웃 이동의 총량을 측정합니다. 사용자가 버튼을 누르려는데 갑자기 광고가 뜨면서 다른 것을 누르게 되는 등의 불편한 경험을 방지합니다. (권장: 0.1 이하)
- **실용적인 최적화 기법:**
    - **이미지 최적화:** WebP, AVIF 같은 차세대 이미지 포맷을 사용하고, `<img>` 태그에 `srcset` 속성을 부여하여 화면 크기에 맞는 이미지를 제공합니다. 화면 밖에 있는 이미지는 `loading="lazy"` 속성으로 지연 로딩합니다.
    - **코드 분할 (Code Splitting):** React의 `React.lazy()`, Vue Router의 동적 import 등을 사용하여 라우트(페이지)별로 JavaScript 번들을 분리합니다. 사용자가 방문하는 페이지에 필요한 코드만 초기에 로드하여 초기 로딩 속도를 개선합니다.
    - **지연 로딩 (Lazy Loading):** 이미지뿐만 아니라, 초기 렌더링에 필수적이지 않은 컴포넌트나 라이브러리도 필요할 때 동적으로 로드합니다.
    - **자원 최소화:** JavaScript, CSS, HTML 코드를 압축(Minification)하고, 사용하지 않는 코드를 제거(Tree Shaking)하여 파일 크기를 줄입니다.
    - **캐싱 활용:** 브라우저 캐시나 CDN을 적극적으로 활용하여 반복적인 리소스 요청을 줄입니다.

---

### **웹 접근성 (WCAG)**

장애 유무와 상관없이 모든 사용자가 웹사이트의 정보에 동등하게 접근하고 이용할 수 있도록 보장하는 것입니다. WCAG(Web Content Accessibility Guidelines)의 4대 원칙을 준수해야 합니다.

- **WCAG 4대 원칙:**
    1. **Perceivable (인식의 용이성):** 모든 콘텐츠는 사용자가 인식할 수 있어야 합니다.
    2. **Operable (운용의 용이성):** 사용자는 UI 요소와 상호작용하고 탐색할 수 있어야 합니다.
    3. **Understandable (이해의 용이성):** 정보와 UI 작동 방식은 이해하기 쉬워야 합니다.
    4. **Robust (견고성):** 콘텐츠는 다양한 사용자 에이전트(브라우저, 보조 기술 등)에서 안정적으로 해석될 수 있어야 합니다.
- **구체적인 구현 예시:**
    - **시맨틱 HTML:** `<div>`와 `<span>`만으로 구조를 짜는 대신, `<nav>`, `<main>`, `<header>`, `<article>`, `<button>` 등 의미에 맞는 태그를 사용합니다. 이는 스크린 리더가 페이지 구조를 정확히 파악하는 데 도움을 줍니다.
    - **ARIA 속성:** 시맨틱 HTML만으로 의미를 전달하기 어려울 때 ARIA(Accessible Rich Internet Applications) 속성을 사용합니다. 예를 들어, 커스텀 디자인된 체크박스에 `role="checkbox"`와 `aria-checked="true"`를 부여하여 스크린 리더 사용자에게 이것이 체크박스임을 알려줍니다.
    - **키보드 내비게이션:** 마우스를 사용할 수 없는 사용자를 위해 모든 인터랙티브 요소(링크, 버튼, 폼 등)가 Tab 키로 접근 가능해야 하며, 현재 포커스된 요소가 시각적으로 명확히(`outline` 스타일) 표시되어야 합니다.
    - **이미지 `alt` 속성:** 모든 의미 있는 이미지에 내용을 설명하는 대체 텍스트(`alt`)를 제공해야 합니다. 장식용 이미지는 `alt=""`로 비워두어 스크린 리더가 무시하도록 합니다.

---

### **유지보수 가능한 코드와 SOLID 원칙**

SOLID는 객체 지향 설계의 5가지 기본 원칙이지만, 컴포넌트 기반의 현대 프론트엔드 개발에도 훌륭하게 적용될 수 있습니다.

- **프론트엔드에서의 SOLID 적용:**
    - **S (Single Responsibility Principle, 단일 책임 원칙):** 하나의 컴포넌트나 함수는 하나의 책임만 가져야 합니다. 예를 들어, `UserProfile` 컴포넌트는 사용자 정보를 보여주는 책임만 지고, 데이터 페칭 및 상태 관리는 `useUserProfile`이라는 커스텀 훅으로 분리합니다.
    - **O (Open/Closed Principle, 개방-폐쇄 원칙):** 컴포넌트는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 합니다. 기존 코드를 직접 수정하는 대신, `props`나 `children`을 활용하여 기능을 확장합니다. 예를 들어, `Card` 컴포넌트 내부에 어떤 내용이든 넣을 수 있도록 `{children}`을 사용하면 `Card` 자체를 수정하지 않고도 다양한 카드를 만들 수 있습니다.
    - **L (Liskov Substitution Principle, 리스코프 치환 원칙):** 부모-자식 관계의 컴포넌트가 있다면, 자식 컴포넌트는 부모 컴포넌트의 역할을 문제없이 대체할 수 있어야 합니다. React에서는 상속보다 합성을 선호하므로, 이 원칙은 "컴포넌트의 prop 인터페이스를 일관되게 유지하라"로 해석할 수 있습니다.
    - **I (Interface Segregation Principle, 인터페이스 분리 원칙):** 컴포넌트가 자신이 사용하지 않는 `prop`을 받도록 강요해서는 안 됩니다. 거대한 `props` 객체 하나를 넘기기보다, 컴포넌트에 필요한 `prop`만 명확하게 정의하여 전달합니다. TypeScript를 사용하면 인터페이스를 통해 이를 강제하기 쉽습니다.
    - **D (Dependency Inversion Principle, 의존관계 역전 원칙):** 상위 수준의 컴포넌트가 하위 수준의 구현 세부사항에 직접 의존해서는 안 됩니다. 대신 둘 다 추상화에 의존해야 합니다. 예를 들어, 컴포넌트가 `axios` API 호출 코드를 직접 포함하는 대신, 추상화된 `apiClient` 객체를 주입받아 사용하거나, 데이터 페칭 로직을 `React Query`나 `SWR` 같은 라이브러리에 위임하는 것이 좋은 예입니다.

### **크로스 브라우저 호환성**

다양한 브라우저(Chrome, Firefox, Safari, Edge 등)와 디바이스에서 애플리케이션이 일관되게 동작하도록 보장하는 것입니다.

- **주요 과제 및 해결 방안:**
    - **JavaScript 호환성:** 최신 JavaScript 문법(ES6+)은 일부 구형 브라우저에서 지원되지 않을 수 있습니다. **Babel**을 사용하여 최신 코드를 구형 브라우저가 이해할 수 있는 ES5 코드로 변환(Transpiling)합니다.
    - **CSS 호환성:** 특정 CSS 속성은 브라우저별로 접두사(`webkit-`, `moz-` 등)가 필요하거나 지원 범위가 다릅니다. **PostCSS**와 **Autoprefixer**를 사용하면 빌드 시점에 필요한 벤더 접두사를 자동으로 추가해 줍니다.
    - **기능 지원 여부 확인:** 개발 전에 [caniuse.com](https://caniuse.com/)이나 [MDN](https://developer.mozilla.org/) 문서를 통해 사용하려는 웹 API나 CSS 속성의 브라우저 지원 범위를 확인하는 습관이 중요합니다.
    - **테스팅:** 모든 브라우저를 직접 설치하여 테스트하기는 어렵습니다. **BrowserStack**이나 **LambdaTest** 같은 크로스 브라우저 테스트 자동화 도구를 사용하여 다양한 환경에서 UI와 기능이 정상적으로 작동하는지 검증합니다.
